# CareerCore — Architecture Decision Record

Decisions are numbered sequentially. Status is one of: **Accepted**, **Superseded**, **Deprecated**.

---

## ADR-001 — Monorepo layout

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
The platform has four distinct deployable units: a FastAPI backend, a Next.js frontend, a CLI tool, and Terraform infrastructure. These could live in separate repositories.

**Decision:**  
Single repository with top-level directories `backend/`, `frontend/`, `cli/`, `infra/`, and `docs/`. Each directory is a deployable unit with its own runtime and Dockerfile.

**Consequences:**  
- One `git clone`, one PR, one CI run. Schema changes and matching frontend type updates land in the same commit.  
- CI takes longer because both stacks run on every PR. Mitigated by running lint/typecheck/test in parallel and only running Docker builds after tests pass.  
- No code sharing between `backend/` and `frontend/` — different language runtimes enforce this.

## ADR-002 — Core stack selection

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
Stack needed to support async file extraction, AI API calls, job queue processing, object storage, and a reactive frontend.

**Decision:**

| Layer | Choice | Rationale |
|-------|--------|-----------|
| Backend API | FastAPI + SQLAlchemy async | Async-native, Pydantic validation, auto OpenAPI docs |
| Database | PostgreSQL 15 | JSONB, full-text search, pgcrypto UUIDs, mature async driver (asyncpg) |
| Cache / broker | Redis 7 | Celery broker, rate-limit counters, future token allowlist |
| Object storage | MinIO | S3-compatible, self-hosted for dev, swap to S3 in prod with no code change |
| Task queue | Celery | File extraction offloaded from request cycle; `acks_late=True` prevents lost tasks |
| AI | Anthropic Claude | Haiku for structured extraction (cheap), Sonnet for generation (quality) |
| Frontend | Next.js 14 App Router | RSC + client components, TanStack Query, Tailwind, shadcn/ui |

**Consequences:**  
All services are containerized in `docker-compose.yml` with health-checked dependencies. Swapping MinIO for S3 in production requires only environment variable changes.

## ADR-003 — UUID primary keys everywhere

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
Standard options are auto-increment integers or UUIDs. Sequential integers make IDOR attacks trivially easy (`/jobs/1`, `/jobs/2`...) and leak record counts.

**Decision:**  
All tables use `UUID` primary keys generated by `gen_random_uuid()` (PostgreSQL) with a matching Python `default=uuid.uuid4`. Both `server_default` and `default` are set: the Python default fires in application code, the server default fires in raw SQL migrations and seed scripts.

**Consequences:**  
UUIDs are not unguessable but are vastly harder to enumerate than integers. They also work across distributed inserts without coordination. Every `get_for_user()` method enforces a double filter `(id == X, user_id == current_user)` — UUIDs alone are not an authorization mechanism.

## ADR-004 — Separate `models/` from `schemas/`

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
Many FastAPI tutorials use a single class for both the ORM model and the Pydantic schema.

**Decision:**  
`app/models/` contains SQLAlchemy ORM classes (database shape). `app/schemas/` contains Pydantic models (API shape). They are never the same class.

**Consequences:**  
- `UserRead` never exposes `password_hash`. `WorkExperienceRead` returns `profile_id: UUID`, not a nested `Profile` object.  
- Eliminates N+1 serialization bugs caused by lazy-loading SQLAlchemy relationships inside Pydantic validators.  
- `app/ai/schemas.py` holds a third category: AI-internal typed contracts between services and the provider layer. These are never serialized to HTTP responses.

## ADR-005 — Service layer between endpoints and models

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
Endpoints could access the database directly.

**Decision:**  
All business logic lives in `app/services/`. Endpoints receive an HTTP request, validate input, delegate to a service, and convert the result to an HTTP response or map domain exceptions to HTTP status codes. Services never import from `app/api/`.

**Consequences:**  
- Business logic is testable without an HTTP server.  
- Services are reusable from CLI commands (Phase 2) and Celery tasks.  
- Domain exceptions (`AuthError`, `BudgetExceededError`) are raised in services; endpoints catch them and return the correct HTTP status.

## ADR-006 — AIProvider as Protocol, not ABC

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
The AI layer needs to be swappable between `MockAIProvider` (tests), `AnthropicProvider` (Phase 1 production), `OpenAICompatibleProvider` (Phase 2), and `OllamaProvider` (Phase 3).

**Decision:**  
`AIProvider` is a `typing.Protocol` with `runtime_checkable`. Any class that implements the required async methods satisfies the protocol without inheriting from it.

**Consequences:**  
- `MockAIProvider` does not import `AIProvider`, breaking potential circular deps.  
- `isinstance(obj, AIProvider)` works at runtime for contract tests.  
- Adding a new provider means implementing methods, not changing the interface or refactoring call sites.

## ADR-007 — Model selection by task type within AnthropicProvider

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
Anthropic charges per token. Tasks vary in complexity.

**Decision:**  
- **Haiku** — `parse_job_description`, `explain_score` (structured extraction, simple output, 12× cheaper than Sonnet)  
- **Sonnet** — `generate_bullets`, `answer_followup`, `generate_recommendations`, `generate_learning_plan` (reasoning quality and writing matter)

**Consequences:**  
Lower per-analysis cost for the common case. Sonnet is reserved for output the user sees and judges. Haiku is used for internal structured data extraction.

## ADR-008 — Deterministic scoring, no LLM

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
Job fit scoring could be delegated to the AI provider, which would produce natural-language reasoning.

**Decision:**  
`ScoringService` is purely deterministic. The formula weights requirements by type:

| Category | Weight |
|----------|--------|
| Skills | 35% |
| Experience | 20% |
| Projects | 20% |
| Tools | 10% |
| Education | 10% |
| Bonus | 5% |

Matched requirements carry evidence links. Missing requirements are enumerated.

**Consequences:**  
- Scoring is exhaustively unit-testable with no mocking.  
- Scores are auditable and explainable: "72/100 because you matched 7/10 skills."  
- `AIProvider.explain_score()` converts the deterministic `ScoreBreakdown` into natural language — the LLM explains what the formula computed, not what it decided.

## ADR-009 — AI cost guard on every call

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
Uncapped AI API usage per user creates unbounded cost exposure.

**Decision:**  
Every AI call follows this contract:
```
AICostService.check_budget(user) → AI call → AICostService.log_call(...)
```
`check_budget` reads today's `AICallLog` rows for the user and raises `BudgetExceededError` before making any API call. `log_call` writes success and failure outcomes alike.

**Consequences:**  
No AI call can succeed without a prior budget check. Failure logging means debugging is possible even when the upstream API is down. Phase 2 will add Redis caching for `(user_id, date) → total_tokens` to avoid querying `ai_call_logs` on every check.

## ADR-010 — JWT auth: Bearer access token + httponly refresh cookie

**Date:** 2026-04-13 (PR #50, #51)  
**Status:** Accepted

**Context:**  
Standard JWT auth options are: both tokens in response body, both tokens in cookies, or split (access in body, refresh in cookie).

**Decision:**  
- `POST /auth/login` returns `access_token` in the JSON response body (Bearer scheme) and sets `refresh_token` as an `httponly`, `samesite=lax`, `secure` (prod only) cookie scoped to `/api/v1/auth`.  
- `POST /auth/refresh` reads the refresh token from the cookie only (no body parameter).  
- Access tokens are short-lived (configurable via `JWT_ACCESS_TOKEN_EXPIRE_MINUTES`).

**Consequences:**  
- The refresh token is never accessible to JavaScript — XSS cannot steal it.  
- The access token is in memory (or local storage if the frontend chooses), short-lived, and not in a cookie — no CSRF exposure on the access token.  
- The cookie path `/api/v1/auth` restricts the cookie to only the auth endpoints.

## ADR-011 — DB-backed refresh token rotation (not Redis allowlist)

**Date:** 2026-04-13 (PR #51)  
**Status:** Accepted

**Context:**  
Without server-side token invalidation, a stolen refresh token grants indefinite access. Redis was the original design for the allowlist (see `auth_service.py` TODO comment).

**Decision:**  
Refresh tokens are stored as SHA-256 hashes in a `refresh_tokens` table (`token_hash`, `user_id`, `expires_at`, `used_at`). On refresh:
1. Hash the presented token and look it up.
2. If not found, already used, or expired → reject.
3. Mark the stored token as `used_at = now()` (rotation — one-time use).
4. Issue a new token pair and insert a new `refresh_tokens` row.

**Consequences:**  
- Token reuse after theft is detected and rejected.  
- No Redis dependency for auth (Redis is still used for Celery and rate limiting).  
- Logout can be implemented by setting `used_at` on all of a user's active tokens.  
- At scale, expired rows accumulate; a periodic cleanup job should prune `refresh_tokens` where `expires_at < now()`.

## ADR-012 — Audit log is append-only, no FK to User

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
Compliance requires a record of all state-changing operations. User accounts can be deleted.

**Decision:**  
- `audit_logs` has no `ON DELETE CASCADE` or `ON DELETE SET NULL` on `user_id`. `user_id` is a nullable `UUID` column, not a foreign key.  
- All writes go through `AuditService.log_event()`. Direct `db.add(AuditLog(...))` in endpoints is prohibited.  
- `log_event` calls `db.flush()`, not `db.commit()` — the caller owns the transaction. A failed operation rolls back its audit entry.

**Consequences:**  
Deleting a user's account preserves their audit history. Compliance reports can always reconstruct "who did what when." Phase 2 target: create a Postgres role with `INSERT`-only permissions on `audit_logs` and use it for the application connection string.

## ADR-013 — Ownership enforcement via double-filter in service layer

**Date:** 2026-04-13 (C-3 fix, PR #53)  
**Status:** Accepted

**Context:**  
`FileService.get_for_user()` was implemented as `db.get(UploadedFile, file_id)` — it accepted a `user_id` parameter but ignored it. Any authenticated user could retrieve any file's metadata and presigned URL by guessing a UUID.

**Decision:**  
Every `get_for_user(user_id, entity_id)` method in the service layer must use:
```python
select(Entity).where(Entity.id == entity_id, Entity.user_id == user_id)
```
The post-hoc ownership check in the endpoint (`record.user_id != current_user.id`) was defense-in-depth masking a broken service contract and was removed after the fix.

**Consequences:**  
Ownership is enforced at the service boundary, not the endpoint boundary. Endpoints that trust the service contract are correct by construction. Any new `get_for_user` implementation that omits the `user_id` filter will be caught in ownership unit tests.

## ADR-017 — Profile schema: UNIQUE FK for 1:1, JSONB vs ARRAY column choice

**Date:** 2026-04-13 (PR #56, issue #12)  
**Status:** Accepted

**Context:**  
The `Profile` model has a 1:1 relationship to `User`. Sub-entity tables (`work_experiences`, `projects`) store both structured AI-extracted bullet lists and flat text tags. Two design questions arose when writing migration `20260413_0003`:

1. How to enforce the 1:1 user–profile relationship at the database level.
2. Whether to use `JSONB` or `ARRAY(String)` for per-row list columns.

**Decision:**

- **1:1 enforcement:** `profiles.user_id` is a FK to `users.id` with a `UNIQUE` constraint. No separate join table.
- **Bullets** (`work_experiences.bullets`, `projects.bullets`): `JSONB`. Ordered list of strings today; may gain element-level metadata (confidence, source offset) in Phase 2 without requiring a new migration.
- **Tag columns** (`skill_tags`, `tool_tags`, `domain_tags`): `ARRAY(String)`. Flat unordered sets of text labels with no anticipated sub-structure.
- **Sub-entity FK delete behavior:** `profile_id` → `CASCADE` (deleting a profile removes all child rows). `source_file_id` → `SET NULL` (deleting an uploaded file severs the source link but preserves the work experience or project record).

**Consequences:**  
- The 1:1 constraint is enforced by the database; application code cannot accidentally create two profiles for one user.  
- `JSONB` gives bullets room to evolve their element schema; `ARRAY(String)` keeps tag columns simple and avoids JSONB overhead for flat data.  
- `SET NULL` on `source_file_id` means file deletion is safe to perform without cascading content loss.  
- `ARRAY` and `JSONB` columns are PostgreSQL-only; tests involving those types must run against PostgreSQL (integration tests), not SQLite.

## ADR-014 — Test strategy: SQLite unit tests, PostgreSQL integration tests

**Date:** 2026-04-13  
**Status:** Accepted

**Context:**  
Running all tests against PostgreSQL requires a live database. Unit tests of pure business logic do not need it.

**Decision:**  
- **Unit tests** (`tests/unit/`) use SQLite in-memory via `aiosqlite`. No external services required. Fast, offline-capable.  
- **Integration tests** (`tests/integration/`) use PostgreSQL via `TEST_DATABASE_URL`. These run in CI with a service container.  
- `AI_PROVIDER=mock` is set at the `conftest.py` level and at the CI workflow level. No test can call a real AI API.  
- FastAPI dependency injection is used for overrides (`app.dependency_overrides`) — the DI system is exercised, not bypassed.

**Consequences:**  
SQLite does not support `JSONB`, `ARRAY`, or some PostgreSQL-specific constraints. Tests involving those types go in `tests/integration/`. The split lets the unit suite run in under 5 seconds with no services.

## ADR-015 — Issue workflow: one issue, one branch, one PR

**Date:** 2026-04-14  
**Status:** Accepted

**Context:**  
Multiple agents and contributors working concurrently need a clear unit of work to prevent collisions and maintain traceability.

**Decision:**  
1. Pick one P1 issue from the backlog.  
2. Assign it to one person (or agent).  
3. Move it to **In Progress**.  
4. Implement in a branch named `issue-{number}-{slug}`.  
5. Open a PR linked to the issue (`Closes #N` in the body).  
6. Move to **Done** when the PR is merged.

No issue is worked on by more than one branch at a time. No branch covers more than one issue. PRs are not merged without at least one passing test covering the new behavior.

**Consequences:**  
Clean git history — every merge commit traces to an issue. Prevents two agents from implementing the same thing in parallel. Enforces that work is always associated with a tracked requirement.

## ADR-016 — Scoring evidence is entity-backed and deterministic

**Date:** 2026-04-13 (PR #58, issue #25)  
**Status:** Accepted

**Context:**  
ADR-008 established that job-fit scoring must be deterministic and must not call an LLM. Issue `#25` forced the next concrete design question: what exactly counts as a full match, what counts as a partial match, and how evidence should be persisted so later layers can explain the score without re-deriving it.

**Decision:**  
- `ScoringService` normalizes requirement text and profile text with case-folding and punctuation stripping before matching.  
- Matching is category-specific:
  - `skill` → `Profile.skills[].name`
  - `tool` → `work_experiences.tool_tags` and `projects.tool_tags`
  - `experience` → `work_experiences.skill_tags` and `description_raw`
  - `project` → `projects.skill_tags`, `description_raw`, and `bullets`
  - `education` → `certifications.name` and `issuer`
- Match classification:
  - **full** = exact normalized match, or corroborated by 2+ distinct profile entities
  - **partial** = non-exact substring/token overlap from 1 profile entity
  - **missing** = no deterministic match found
- Every non-missing match writes `MatchedRequirement` rows with `source_entity_type`, `source_entity_id`, and confidence. Missing requirements write `MissingRequirement` rows.
- `ScoreBreakdown.evidence_map` stores the same linkage at the output layer, keyed by requirement ID.

**Consequences:**  
- Scores are reproducible and traceable back to concrete profile rows.  
- Resume generation and score explanation can cite stored evidence instead of asking an LLM to rediscover it.  
- The heuristic stays simple and auditable, but it will under-match some semantic equivalents until richer requirement storage or synonym handling is added in a later phase.

## ADR-018 — Profile completeness is a persisted deterministic weighted score

**Date:** 2026-04-13 (PR #60, issue #16)  
**Status:** Accepted

**Context:**  
`profiles.completeness_pct` existed as a stored column but the service layer still returned a placeholder value. The application needed a real completeness signal for the Phase 1 profile model, and that signal had to stay in sync when the user edited either the root profile fields or any of the Phase 1 child sections.

**Decision:**  
Profile completeness is computed deterministically in `ProfileService` using these Phase 1 weights:

| Section | Weight |
|---------|--------|
| `display_name` present | 10% |
| `current_title` present | 10% |
| `target_domain` present | 10% |
| At least 1 work experience | 25% |
| At least 1 skill | 20% |
| At least 1 project | 15% |
| At least 1 certification | 10% |

The score is persisted to `profiles.completeness_pct` whenever:
- a profile is created
- the root profile is updated
- a work experience, skill, project, or certification is created, updated, or deleted

Whitespace-only string fields do not count as present. No AI call is involved.

**Consequences:**  
- Completeness is stable, auditable, and unit-testable because the score is derived from current DB state rather than AI output.  
- The stored value can be returned directly to clients without recomputing the formula in frontend code.  
- Any future Phase 2 profile section must explicitly decide whether it changes the completeness formula; adding a new table alone does not affect the score.  
- Endpoints mutating profile sub-entities now own the responsibility to trigger a completeness recalculation before the request transaction commits.

## ADR-019 — Route protection is default-deny; public exceptions are explicit

**Date:** 2026-04-13 (issue #10)  
**Status:** Accepted

**Context:**  
The API already used `get_current_user` on most non-public endpoints, but the rule was implicit and the behavior for missing access tokens depended on FastAPI's default `HTTPBearer`, which returned `403` for absent credentials. The application also exposed `/docs`, `/redoc`, and `/openapi.json` in all environments.

**Decision:**  
- Every non-public API endpoint must declare `get_current_user` explicitly as a FastAPI dependency.  
- Missing, invalid, and expired Bearer access tokens all return `401 Unauthorized` with `WWW-Authenticate: Bearer`.  
- The route-protection audit is enforced with tests over the endpoint source tree, so the public exception set stays reviewable even while the shared app bootstrap remains noisy in local tests.  
- The public API exceptions are:
  - `GET /health`
  - `POST /api/v1/auth/register`
  - `POST /api/v1/auth/login`
  - `POST /api/v1/auth/refresh`
- Documentation routes are public only in development:
  - `GET /docs`
  - `GET /redoc`
  - `GET /openapi.json`
  In production, those routes are disabled.

**Consequences:**  
- Route protection can be audited mechanically: anything not in the public exception set must depend on `get_current_user`.  
- Clients get the correct auth semantics for missing and expired access tokens; `403` is reserved for authorization failures after authentication.  
- The auth boundary stays testable even when full app-import integration tests are temporarily constrained by import-time settings and database bootstrap coupling.  
- Production deployments no longer expose the interactive API docs or OpenAPI schema unless that policy is changed deliberately.
